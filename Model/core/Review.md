# Review — System Feedback Loop

The system without self-correction is just a plan. This file is the maintenance protocol.

---

## Weekly Review (每周日，30 min)

Pick 5 signals. Log them. Ask one question per layer.

| Layer | Signal | Trigger for action |
| ----- | ------ | ------------------ |
| Hardware | Sleep avg (hrs), workouts completed | < 7h sleep 3+ nights → deprioritize everything else next week |
| Resource | Savings rate, quant system P&L | Net negative 2 weeks → pause active strategy, diagnose |
| Mind | Hours of deep work, one thing produced | 0 output 2 weeks → check if wrong mental mode is running |
| Schedule | Energy-action alignment | Doing low-energy tasks in high-energy slots → reschedule |

**One question to close the review:**
> What is the single biggest drag on system performance this week? What parameter changes next week?

---

## Monthly Review (每月最后一天，2 hrs)

Run a full layer health check.

### Hardware Layer
- Strength trend (progressive overload happening?)
- Energy quality (consistent high-energy mornings?)
- Any recurring physical signals of overload (joint pain, persistent fatigue)?

### Resource Layer
- Net worth delta vs last month
- Quant system: is it running as designed or drifting?
- Cash reserve: still above minimum threshold?

### Mind Layer
- What did I produce this month? (Code / Writing / Video / Building / Mentoring)
- Which output compounded back into the system? (e.g., writing built network, code built career capital)
- Am I in the right 20% exploration domain? Is it still generative?

### System-wide
- Which inter-system dependency is currently the weakest link?
- Is the slow practice principle actually applied, or did I rush?

**One output from monthly review:**
> Write one paragraph: what changed, what parameter was adjusted, and why. Saved in `/investment/daily/` or an `ops/log/` if created.

---

## Parameter Change Protocol

Not every signal warrants a change. Use this to avoid over-tuning:

1. Signal appears → log it (don't act yet)
2. Signal persists 2+ consecutive review cycles → investigate root cause
3. Root cause identified → change exactly one parameter
4. Run 4 weeks → measure effect before changing another

This is the "one variable at a time" rule applied to life systems.

---

## Shutdown Criteria

Some things should trigger a hard system halt and reset, not a parameter tweak:

- Sleep average drops below 6h for 2+ weeks → halt all non-essential projects
- Financial reserve hits minimum threshold → freeze all active strategy risk
- Mental exhaustion (inability to enter deep work for 1+ week) → mandatory 3-day GC cycle
